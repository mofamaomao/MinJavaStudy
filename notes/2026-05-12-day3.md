# 2026-05-12 学习笔记（第3天）

## 今日主题
ECS架构 — Mindustry的组件化实体系统

## 学习阶段
阶段1 · 第3天

## 核心概念（我的总结）

### ECS是什么
- **E**ntity（实体）= 只是一个容器，本身没有数据
- **C**omponent（组件）= 独立的数据/能力模块
- **S**ystem（系统）= 批量处理某类组件的逻辑

### Mindustry的ECS变体
纯ECS中Component只有数据，但Mindustry做了折中：
- Component自带简单逻辑方法
- 避免了System和Component之间过度耦合
- 更实用，但不是教科书级别的纯ECS

### Component层（C）
`core/src/mindustry/entities/comp/` 目录下大量 `*Comp.java`：
- `HealthComp`：血量相关
- `ItemsComp`：物品存储
- `HitboxComp`：碰撞箱
- `BuildingComp`：建筑运行时实例
- 等等...

### Entity层（E）
- `annotations/EntityProcess.java` 在**编译期**读取所有Comp
- 自动生成组合了多个Comp的实体类（`Unit`、`Building`等）
- 生成的代码在 `gen/` 目录，不在源码里
- 带来的复杂度：调试困难，理解门槛高

### System层（S）
- `GroupDefs.java` 用 `@GroupDef` 注解定义实体分组
  - `G.unit` = 所有拥有 `Unitc` 组件的实体
  - `G.build` = 所有建筑实体
  - `G.bullet` = 所有子弹实体
- `Logic.update()` 每帧调用 `Groups.update()`
- `Groups.update()` 遍历所有组，批量调用每个实体的 `update()`

### Block vs BuildingComp（复习）
- `Block` = 编译期模板（全局一份）
- `BuildingComp` = 运行时实例（每个放置的建筑一份）
- 存档保存的是实例的字段值，不是类本身

### 组合优于继承
- 不通过继承获得能力，而是把模块作为字段包含进来
- `@Nullable ItemModule items` → 需要时才创建，节省内存
- ECS把这个思想推向极致

## 关联源码
- `core/src/mindustry/entities/comp/` → 所有Comp定义
- `core/src/mindustry/entities/GroupDefs.java` → 实体分组定义
```java
  @GroupDef(value = Unitc.class, spatial = true, mapping = true) G unit;
  @GroupDef(value = Buildingc.class) G build;
  @GroupDef(value = Bulletc.class, spatial = true, collide = true) G bullet;
```
- `annotations/src/main/java/mindustry/annotations/entity/EntityProcess.java`
  → 编译期代码生成
- `core/src/mindustry/core/Logic.java` → `update()` 方法
```java
  Groups.update(); // 每帧遍历所有组，批量更新所有实体
```
